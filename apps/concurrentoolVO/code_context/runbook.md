## Deploy

### Current state (pre-monorepo-migration)

Vercel auto-deploys on push to `main` of the upstream GitHub repo `Gamingsixtie/concurrentie-rekentool-VO`. App-Dev OS subtree-import is local only — production is unaffected.

### Standard flow

1. Work on `feature/*` branch (in upstream repo) → Vercel preview deploy auto-generated
2. Test in preview, validate
3. PR → merge to `main` → Vercel production deploy auto-triggered (~2 min)
4. Verify production deploy via the URL above + key user flow (school overview → wizard → comparison)

> **Don't** push directly to `main` — `branch-guard` hook (App-Dev OS) hard-blocks. Always PR. Don't auto-push (overrides rekentool's pre-existing CLAUDE.md rule, see `apps/concurrentoolVO/AGENTS.md` § App-specific overrides).

## Rollback

### Path 1 — Vercel "Promote previous" (~30s, no git)

1. Open Vercel dashboard → concurrentoolVO project → Deployments
2. Find last known-good deployment (green checkmark + production tag)
3. Click "..." → "Promote to Production"
4. Verify within ~30s the production URL serves the old version

This is the **first response** when production is broken. Bleeds the wound, no git involved.

### Path 2 — source-sync via revert PR (within 24h)

After Path 1 promotes the old version, open a new branch in the upstream repo:

```bash
git checkout -b revert/<short-name> main
git revert <bad-sha>
git push -u origin revert/<short-name>
gh pr create --base main --head revert/<short-name>
```

Merge the revert PR. This re-aligns git history with what's actually deployed. Skipping Path 2 means git drifts from production, and the next deploy could re-promote the broken commit.

### Database rollback

Supabase migrations are append-only — never edit a merged migration. To "rollback" a schema change: write a new migration that compensates (e.g., drop the column, restore the old column with backfill). See § Database operations.

- Never commit any of these to git (already gitignored: `.env`, `.env.local`)

## Database operations

### Migrations

```bash
npx supabase migration new <name>   # create new migration file in supabase/migrations/
npx supabase db reset                # local: drop + reapply all migrations (destructive on local)
npx supabase db push                 # apply pending migrations to remote dev project
```

**Rules:**
- Migrations are immutable after merge to main — never edit a merged migration
- Destructive change (drop column, rename, type change)? 5-step pattern:
  1. Add new column / table
  2. Backfill from old → new
  3. Deploy + verify reads via dual-read
  4. Switch writes to new
  5. Remove old (next migration)

### Backups

Supabase plan determines retention:
- Free tier: 7-day backup retention
- Pro tier: 7-day PITR (point-in-time recovery) + 30-day daily snapshots

**Quarterly**: do a restore-test on the dev project to verify the backup is actually restorable. Don't trust a backup you've never restored.

### RLS

All tables have RLS enabled (migration 002 + fixes in 006/007). When adding a new table:
1. Migration creates the table
2. Same migration enables RLS (`ALTER TABLE ... ENABLE ROW LEVEL SECURITY;`)
3. Same migration adds explicit policies (or denies all by default — be explicit)

Storage `documents` bucket also has RLS (migrations 003 + 007).

## Disaster scenarios

| Scenario | First response | Follow-up |
|---|---|---|
| Anthropic API down | App-side: catch and show Dutch fallback message ("AI-intake niet beschikbaar"). Wizard still works manually. | Monitor `status.anthropic.com`. No code change needed if upstream. |
| Supabase down (data layer) | Service worker serves cached reads from IndexedDB. Writes queue via `offline-queue.ts`. | Monitor `status.supabase.com`. App degrades gracefully. |
| Vercel platform down | Tool fully unavailable (no fallback). | Monitor `vercel-status.com`. Wait + apologize via known channels. |
| Bad code shipped | Vercel "Promote previous" (Path 1 above) | Open revert PR (Path 2) within 24h |
| Bad migration shipped to prod | Restore from latest backup (Pro tier PITR), OR write compensating migration | Test recovery on dev first if not in fire |
| `default-prices.ts` accidentally edited | `git revert` the commit + redeploy | Verify production prices match expected |
| Supabase RLS regression | Read recent migration; if RLS was modified, restore via new migration | Re-enable RLS + add explicit policy |
| API key leaked | Rotate immediately at the upstream provider (Anthropic / Supabase / Sentry) → update Vercel env → redeploy | Scan git history for exposure scope |
