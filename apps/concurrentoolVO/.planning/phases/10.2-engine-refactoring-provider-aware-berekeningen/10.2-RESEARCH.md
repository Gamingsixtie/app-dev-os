# Phase 10.2: Engine Refactoring -- Provider-aware berekeningen - Research

**Researched:** 2026-03-24
**Domain:** TypeScript engine architecture, provider-specific price calculation, strategy pattern
**Confidence:** HIGH

## Summary

Phase 10.2 refactors the calculation engine from a naive "flat price x students" model to provider-aware calculators that understand each provider's actual pricing model. The current `calculateComparison()` in `src/engine/price-comparison.ts` treats every provider identically: look up a `PriceRecord`, multiply `amountPerStudent * totalStudents`. Provider-specific logic (JIJ tier selection, DIA package optimization, Cito bundle pricing) is scattered across the Zustand store (`store.ts`) as pre-processing steps (`applyDynamicJijPrices`, `applyDiaVolumeDiscount`, `applyCitoBundlePrices`) that mutate the price array before feeding it to the engine.

The Phase 10.1 foundation is complete: `PricingStrategy` discriminated union (flat/tiered-license/package-bundle/platform+module), `PROVIDER_CONFIGS` map with typed configs per provider, and `MODULE_CATALOG` expanded to 10 modules. This phase moves the scattered provider logic INTO pure calculator functions, one per provider, that consume the typed `PricingStrategy` configs. The result type needs a `breakdown` field per module explaining HOW the price was computed (not just the final number).

The five success criteria require: (1) per-provider `ProviderPriceCalculator` implementations, (2) `calculateComparison()` produces breakdown per module, (3) JIJ prices vary with school size (tier selection), (4) DIA package optimization integrated, (5) school-specific overrides from Supabase take precedence. All logic must remain pure functions; the store becomes a thin orchestrator.

**Primary recommendation:** Create a `ProviderPriceCalculator` interface with a `calculate(moduleId, totalStudents, override?)` method per provider. Implement 4 concrete calculators using switch-on-strategy-type pattern. Refactor `calculateComparison()` to use these calculators and return an enriched `ModuleComparison` with a `breakdown` field. Move store pre-processing into the engine layer.

### Pattern 1: ProviderPriceCalculator Interface
**What:** A common interface for all provider-specific price calculators
**When to use:** Each provider implements this to compute module costs based on its pricing model
**Example:**
```typescript
// src/engine/calculators/types.ts

export interface PriceBreakdownStep {
  label: string;          // Dutch explanation, e.g. "Licentie 3: EUR 975/jaar / 800 lln"
  amount: number;         // Euro amount for this step
}

export interface ModulePriceResult {
  pricePerStudent: number;
  totalCost: number;
  breakdown: PriceBreakdownStep[];
  isPackagePrice: boolean;
  packageId?: string;
  tierId?: number;
}

export interface ProviderPriceCalculator {
  /**
   * Calculate the cost for a single module.
   * Pure function: no side effects.
   *
   * @param moduleId - The module to calculate
   * @param totalStudents - School size (affects tier selection, volume discounts)
   * @param overridePrice - School-specific price override (takes precedence)
   */
  calculateModule(
    moduleId: string,
    totalStudents: number,
    overridePrice?: number,
  ): ModulePriceResult | null;

  /**
   * Calculate all selected modules together (enables package optimization).
   * Pure function: no side effects.
   */
  calculateAll(
    selectedModuleIds: string[],
    totalStudents: number,
    overridePrices?: Map<string, number>,
  ): Map<string, ModulePriceResult>;
}
```

### Pattern 2: Strategy-Based Calculator Factory
**What:** Factory function that creates the right calculator based on PricingStrategy type
**When to use:** When `calculateComparison()` needs a calculator for each provider
**Example:**
```typescript
// src/engine/calculators/index.ts
import type { ProviderConfig } from '@/data/providers';
import type { ProviderPriceCalculator } from './types';
import { CitoCalculator } from './cito-calculator';
import { DiaCalculator } from './dia-calculator';
import { JijCalculator } from './jij-calculator';
import { FlatCalculator } from './flat-calculator';

export function createCalculator(config: ProviderConfig): ProviderPriceCalculator {
  switch (config.pricingStrategy.type) {
    case 'platform+module':
      return new CitoCalculator(config);
    case 'package-bundle':
      return new DiaCalculator(config);
    case 'tiered-license':
      return new JijCalculator(config);
    case 'flat':
      return new FlatCalculator(config);
    default: {
      const _exhaustive: never = config.pricingStrategy;
      throw new Error(`Unknown pricing strategy: ${(_exhaustive as { type: string }).type}`);
    }
  }
}
```

### Anti-Patterns to Avoid
- **Breaking the existing `calculateComparison()` signature immediately:** The store and possibly other consumers call it. Refactor the internals first, then update the signature, then update consumers.
- **Putting provider logic in the store:** The entire point is to move provider-specific pricing logic INTO the engine calculators, OUT of the store.
- **Stateful calculators:** Calculators must be pure functions (or stateless classes). They receive all inputs as parameters.
- **Changing price values:** This is a refactoring -- numerical output must be identical for same inputs.

| Cito bundle pricing | New bundle logic | Reuse existing `applyCitoBundlePrices` logic from cito-bundles.ts | Already tested, handles partial bundle selection |

## Common Pitfalls

### Pitfall 1: Numerical Parity Regression
**What goes wrong:** After refactoring, the engine produces slightly different totals due to rounding differences or different order of operations.
**Why it happens:** The current pipeline applies DIA volume discount BEFORE package selection; the refactored calculator might do it in different order.
**How to avoid:** Write a parity test that compares old pipeline output vs. new calculator output for multiple school sizes (100, 500, 1000+ students). Assert exact match on totals.
**Warning signs:** Existing price-comparison tests fail; DIA totals differ by small amounts.

### Pitfall 2: Circular Import Between Calculators and Engine
**What goes wrong:** Calculator files import from engine (types), engine imports from calculators.
**Why it happens:** `ProviderCost` is defined in `price-comparison.ts`, calculators return it, `calculateComparison` uses calculators.
**How to avoid:** Define all shared types (ProviderCost, PriceBreakdown, ModulePriceResult) in a separate types file (`calculators/types.ts`). `price-comparison.ts` imports from there. Calculators import from there. No circular dependency.
**Warning signs:** "Cannot access before initialization" errors; build fails.

### Pitfall 3: Store Override Logic Not Preserved
**What goes wrong:** School-specific price overrides (from Supabase) stop working because the new engine doesn't know about them.
**Why it happens:** Currently overrides are merged into the price array in the store (`mergeOverrides`). The new calculators accept `overridePrices` but the store forgets to pass them.
**How to avoid:** Success criterion 5 explicitly requires Supabase overrides to take precedence. The calculator's `overridePrice` parameter is the mechanism. The store must pass active school prices as overrides.
**Warning signs:** School-specific prices not reflected in comparison.

### Pitfall 4: DIA Package Result Not Propagated
**What goes wrong:** The `diaPackageResult` (which package was selected, savings amount) is lost because the DIA calculator handles packages internally but doesn't expose the result.
**Why it happens:** Currently `computeExtendedResults()` runs `calculateDiaModuleCosts()` separately and stores the result. After refactoring, package selection happens inside the DIA calculator.
**How to avoid:** The DIA calculator's `calculateAll()` must expose the `DiaPackageResult` (or equivalent) so the store/UI can show package info. Either return it in a side-channel or include it in the `ModulePriceResult` metadata.
**Warning signs:** DIA package badge/explanation disappears from UI.

### Pitfall 5: Breaking Hybrid Scenario and Sensitivity Analysis
**What goes wrong:** `calculateHybridScenario()` and `calculateSensitivity()` consume `ComparisonResult` and assume the current shape. If `ProviderCost` changes shape (new fields), they might break.
**Why it happens:** Adding fields to `ProviderCost` is additive and non-breaking, but removing or renaming fields would break downstream engines.
**How to avoid:** Only ADD new fields to `ProviderCost` (breakdown, isPackagePrice, tierId). Never remove existing ones (pricePerStudent, totalCost, studentCount, priceRecord). Run all engine tests after refactoring.
**Warning signs:** Failing hybrid-scenario or sensitivity tests.

### Pitfall 6: Contract Period Multiplication Happens Twice
**What goes wrong:** Cito contract period discounts are applied inside the calculator AND again in `applyContractPeriodToResult()`.
**Why it happens:** Currently contract period logic is in `applyContractPeriodToResult()` which multiplies all totals. If the Cito calculator also applies contract pricing, it doubles.
**How to avoid:** Clear separation: calculators compute ANNUAL prices. Contract period multiplication stays as a post-processing step (existing `applyContractPeriodToResult`). Calculators should NOT apply contract period logic.
**Warning signs:** Cito prices are suspiciously low (doubled discount).

## Code Examples

### JIJ Calculator (school-size-dependent pricing)
```typescript
// src/engine/calculators/jij-calculator.ts
import type { JijProviderConfig } from '@/data/providers/jij';
import type { ProviderPriceCalculator, ModulePriceResult, PriceBreakdownStep } from './types';

export class JijCalculator implements ProviderPriceCalculator {
  constructor(private config: JijProviderConfig) {}

  calculateModule(
    moduleId: string,
    totalStudents: number,
    overridePrice?: number,
  ): ModulePriceResult | null {
    const defaultPrice = this.config.defaultPrices.find(
      p => p.moduleId === moduleId && p.provider === 'jij',
    );
    if (!defaultPrice) return null;

    // Override takes precedence (success criterion 5)
    if (overridePrice !== undefined) {
      return {
        pricePerStudent: overridePrice,
        totalCost: overridePrice * totalStudents,
        breakdown: [
          { label: 'Schoolspecifieke prijs (overschrijving)', amount: overridePrice },
          { label: `${totalStudents} leerlingen x EUR ${overridePrice.toFixed(2)}`, amount: overridePrice * totalStudents },
        ],
        isPackagePrice: false,
      };
    }

    // JIJ! has license + per-test model: price varies with school size
    // Modules with amountPerStudent === 0 are included in license (no extra cost)
    if (defaultPrice.amountPerStudent === 0) {
      return {
        pricePerStudent: 0,
        totalCost: 0,
        breakdown: [{ label: 'Onderdeel van JIJ! licentie (geen meerprijs)', amount: 0 }],
        isPackagePrice: false,
      };
    }

    const { tiers, defaultTestsPerStudent } = this.config.pricingStrategy;
    const totalAdministrations = totalStudents * defaultTestsPerStudent;

    // Select tier based on total administrations
    const tier = tiers.find(
      t => totalAdministrations >= t.minAdministrations && totalAdministrations <= t.maxAdministrations,
    ) ?? tiers[tiers.length - 1];

    const licenseCostPerStudent = tier.annualFee / totalStudents;
    const testCostPerStudent = defaultTestsPerStudent * tier.pricePerTest;
    const pricePerStudent = Math.round((licenseCostPerStudent + testCostPerStudent) * 100) / 100;
    const totalCost = pricePerStudent * totalStudents;

    const breakdown: PriceBreakdownStep[] = [
      { label: `${tier.label}`, amount: tier.annualFee },
      { label: `Licentie: EUR ${tier.annualFee.toFixed(2)} / ${totalStudents} lln = EUR ${licenseCostPerStudent.toFixed(2)}/lln`, amount: licenseCostPerStudent },
      { label: `Toetskosten: ${defaultTestsPerStudent} afnames x EUR ${tier.pricePerTest.toFixed(2)} = EUR ${testCostPerStudent.toFixed(2)}/lln`, amount: testCostPerStudent },
      { label: `Totaal per leerling: EUR ${pricePerStudent.toFixed(2)}`, amount: pricePerStudent },
    ];

    return {
      pricePerStudent,
      totalCost,
      breakdown,
      isPackagePrice: false,
      tierId: tier.tier,
    };
  }

  calculateAll(
    selectedModuleIds: string[],
    totalStudents: number,
    overridePrices?: Map<string, number>,
  ): Map<string, ModulePriceResult> {
    const results = new Map<string, ModulePriceResult>();
    for (const moduleId of selectedModuleIds) {
      const override = overridePrices?.get(moduleId);
      const result = this.calculateModule(moduleId, totalStudents, override);
      if (result) results.set(moduleId, result);
    }
    return results;
  }
}
```

### Store Changes

The store's `initialize()` and `recalculate()` currently run a 6-step pipeline:
1. `applyDynamicJijPrices(DEFAULT_PRICES, studentCounts)` -- MOVES to JIJ calculator
2. `applyDiaVolumeDiscount(...)` -- MOVES to DIA calculator
3. `applyCitoBundlePrices(...)` -- MOVES to Cito calculator
4. `calculateComparison(selectedModules, studentCounts, dynamicPrices)` -- REFACTORED to use calculators
5. `applyContractPeriodToResult(...)` -- STAYS (post-processing)
6. `computeExtendedResults(...)` -- SIMPLIFIED (DIA package result now from calculator)

After refactoring, the store calls:
1. `calculateComparison(selectedModules, studentCounts, { citoBundleType, overrides })` -- NEW signature
2. `applyContractPeriodToResult(...)` -- UNCHANGED
3. Sensitivity and hybrid calculations -- UNCHANGED (consume enriched result)

## Open Questions

1. **calculateComparison() Signature Change Strategy**
   - What we know: The function currently takes `(selectedModules, studentCounts, prices: PriceRecord[])`. The new version needs `citoBundleType` and `overrides` instead of pre-processed prices.
   - What's unclear: Should the old signature be kept temporarily with a deprecation wrapper, or changed in one go?
   - Recommendation: Change in one go. The only consumers are `store.ts` (initialize + recalculate) and tests. Both are updated as part of this phase.

2. **Breakdown Label Language**
   - What we know: CLAUDE.md says UI text in Dutch. Breakdown labels will be shown in ModuleDetailPanel.
   - What's unclear: Are breakdown labels "UI text" (Dutch) or "data" (English)?
   - Recommendation: Dutch -- these are user-facing explanations displayed in the "Berekeningsformule" section. The existing `sourceLabel` field is already Dutch.

3. **DIA Package Result Exposure**
   - What we know: The store currently has `diaPackageResult` as a separate state field used by `ModuleDetailPanel` to show package info.
   - What's unclear: Should the `DiaPackageResult` be returned as part of `ComparisonResult` or separately?
   - Recommendation: Add `diaPackageResult: DiaPackageResult | null` to `ComparisonResult`. The DIA calculator can compute this internally and `calculateComparison()` can surface it. This eliminates the separate `calculateDiaModuleCosts()` call in the store.
