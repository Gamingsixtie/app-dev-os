# Phase 10.3: UX Overhaul -- Wizard & Vergelijking - Research

**Researched:** 2026-03-24
**Domain:** React UI components, Tailwind CSS 4, Zustand state management
**Confidence:** HIGH

## Summary

This phase is a UI-heavy enhancement of existing components. The codebase already has WizardStep3.tsx (module selection), ComparisonTable.tsx (comparison view), ModuleDetailPanel.tsx (expandable details), and CitoBundleSelector.tsx. The engine layer from Phase 10.2 already provides `PriceBreakdownStep[]` per module per provider and the `MODULE_CATALOG` (10 modules with `availableFrom: ProviderKey[]`) is in place.

The work is primarily component refactoring and new UI elements: extending WizardStep3 with provider badges and quick-picks, making ComparisonTable columns dynamic (user-selectable providers), adding collapsible pricing model explanation cards per provider, and wiring the existing engine breakdown data into the ModuleDetailPanel with a step-by-step calculation display.

**Primary recommendation:** Extend existing components in-place (not rewrite), using the established Tailwind + Zustand patterns. Add `visibleProviders: ProviderKey[]` to the Zustand store for dynamic column selection. All new UI text must be in Dutch.

**What:** Store `visibleProviders: ProviderKey[]` in Zustand store. Default = `['cito']` + providers detected from school's productgebruik. Checkbox row above table toggles providers.
**When to use:** ComparisonTable, ModuleDetailPanel column rendering.

### Anti-Patterns to Avoid
- **Creating new React Context:** CLAUDE.md forbids it. Use Zustand store for visibleProviders state.
- **Hard-coding 3 provider columns:** Must be dynamic based on visibleProviders array.
- **Duplicating engine logic in UI:** The breakdown labels and amounts come from the engine. Do not recompute in components.
- **Prop drilling provider selection:** Put it in the Zustand store, not passed through 5 component levels.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Price breakdown calculation | Custom UI-side math | Engine's PriceBreakdownStep[] | Engine already does this with Dutch labels |
| Module grouping by category | Manual filtering | MODULE_CATALOG + MODULE_CATEGORIES | Already structured with category field |
| Provider availability | Manual lookup tables | ModuleDefinition.availableFrom | Single source of truth in MODULE_CATALOG |
| Pricing model descriptions | Inline strings | PROVIDER_CONFIGS[key].pricingStrategy | Already defined per provider config |
| Collapse/expand animation | Custom JS animation | Tailwind transition classes or native `<details>` | Simpler, accessible, fewer bugs |

## Common Pitfalls

### Pitfall 1: Stale Provider Columns After Bundle Change
**What goes wrong:** User changes Cito bundle type, but ComparisonTable doesn't re-render with updated breakdown.
**Why it happens:** The `citoBundleType` is in the store but column rendering might not react to it.
**How to avoid:** ComparisonTable must subscribe to `citoBundleType` from the store and trigger recalculation. The store's `recalculate()` already handles this.
**Warning signs:** Bundle toggle changes but prices don't update.

### Pitfall 2: Provider Checkbox Default Not Matching School Data
**What goes wrong:** Default visible providers don't include providers the school currently uses.
**Why it happens:** School's moduleSetups with currentProvider need to be read at initialization.
**How to avoid:** Initialize `visibleProviders` from school profile's module setups + always include 'cito'.
**Warning signs:** School uses DIA but DIA column is hidden by default.

### Pitfall 3: Empty Columns for Providers Without Module Coverage
**What goes wrong:** User enables SAQI provider but most modules show "n.v.t." -- table looks empty.
**Why it happens:** SAQI only covers sociaal-emotioneel. Showing an entire column for 1 module is noisy.
**How to avoid:** Show a count badge on the provider checkbox ("SAQI (1 module)") so user knows coverage. The table already filters providers with no data per `activeProviders` logic.
**Warning signs:** Wide table with mostly empty cells.

### Pitfall 4: "Moderne Vreemde Talen" Subcategory Rendering
**What goes wrong:** Frans/Duits/Spaans not visually grouped within "Overige instrumenten".
**Why it happens:** MODULE_CATALOG has them as individual items in `overige-instrumenten` -- no subcategory field exists.
**How to avoid:** Use hardcoded `MVT_MODULE_IDS = ['frans', 'duits', 'spaans']` constant for subcategory grouping in the UI only. Don't add complexity to the data model.
**Warning signs:** Three separate cards without visual grouping.

### Pitfall 5: Inline Bundle Selector Interaction Conflict
**What goes wrong:** Moving CitoBundleSelector into column header creates tight space and may conflict with column sort or expand actions.
**Why it happens:** Column headers have limited space, especially on tablet.
**How to avoid:** Use a compact toggle (segmented control style) rather than a full dropdown. Keep it above the price cells, not inside them.
**Warning signs:** Buttons too small to tap on tablet.

### Provider Selection Checkbox Row
```typescript
function ProviderSelector() {
  const { visibleProviders, toggleProvider } = usePriceComparisonStore();
  const allProviders: ProviderKey[] = ['cito', 'dia', 'jij', 'saqi'];

  return (
    <div className="flex gap-3 mb-4">
      {allProviders.map(p => (
        <label key={p} className="flex items-center gap-1.5 text-sm">
          <input
            type="checkbox"
            checked={visibleProviders.includes(p)}
            onChange={() => toggleProvider(p)}
            disabled={p === 'cito'} // Cito always visible
          />
          <span className={`w-2.5 h-2.5 rounded-full ${PROVIDER_COLORS[p]}`} />
          {PROVIDER_LABELS[p]}
        </label>
      ))}
    </div>
  );
}
```
