Per D-11: comprehensive security audit.

Note: This task reads many files across src/ to audit them. The primary output is src/lib/security-audit-report.md. If CRITICAL/HIGH findings are discovered, the affected source files will also be modified — likely candidates include src/lib/ai-intake.ts (API key handling), src/lib/supabase/client.ts (client config), vercel.json (CSP adjustments), and potentially any file using dangerouslySetInnerHTML or URL injection patterns. The exact files modified depend on audit findings.

1. **XSS scan**: search entire `src/` for:
   - `dangerouslySetInnerHTML` usage — verify all inputs are sanitized
   - `innerHTML` assignments
   - URL injection vectors (user input in `href`, `src`, `window.location`)
   - `eval()`, `new Function()`, `document.write()`

2. **Sensitive data exposure**: check for:
   - API keys in client bundle: grep for `VITE_ANTHROPIC_API_KEY` usage patterns — verify it's only used in serverless functions or with appropriate warnings
   - Supabase anon key exposure (acceptable for RLS-protected tables)
   - Any hardcoded secrets or credentials
   - Source maps in production (acceptable if Sentry is configured)

3. **Input validation audit**: verify:
   - All form inputs have Zod schema validation
   - AI intake free text is not used in unsafe ways (e.g., not injected into HTML)
   - File upload (PDF price extraction) validates file type and size

4. **Dependency vulnerability triage**: run `npm audit --omit=dev` and assess:
   - For each HIGH/CRITICAL finding: is it exploitable in this SPA context?
   - Fix what's fixable via `npm audit fix`
   - Document unfixable/false-positive findings in the audit report

5. **CSP header review**: verify the CSP in vercel.json (from Plan 02) covers:
   - All Supabase domains in connect-src
   - Sentry domains in script-src and connect-src
   - No unnecessary `unsafe-eval` or overly broad wildcards

6. Create `src/lib/security-audit-report.md` documenting:
   - Date of audit
   - Each check performed
   - Findings (with severity: CRITICAL/HIGH/MEDIUM/LOW/INFO)
   - Remediation applied or reason for accepting risk
   - Remaining known risks

7. Fix any CRITICAL or HIGH severity findings directly in the code.

  <action>
Per D-12 (performance profiling):

1. **Build size analysis**:
   - Run `npm run build` and check output for chunk sizes
   - Identify any chunks > 500KB that could benefit from code splitting
   - Check for unnecessary imports that inflate bundle (e.g., importing entire library when only one function needed)

2. **Lazy loading audit**:
   - Verify route-level code splitting is in place (React.lazy or TanStack Router lazy)
   - Check if heavy dependencies (recharts, @react-pdf/renderer, @anthropic-ai/sdk) are lazy-loaded
   - If not lazy-loaded, add dynamic imports for:
     - PDF generation (only loaded when user exports)
     - Recharts (only loaded when viewing charts)
     - AI SDK (only loaded when using intake)

3. **React performance patterns**:
   - Scan for missing React.memo on frequently re-rendered list item components
   - Check for expensive computations without useMemo
   - Verify store selectors are granular (not subscribing to entire store when only one field needed)

4. **Coverage threshold update**:
   - Run `npx vitest run --coverage` to get actual coverage numbers
   - Update vitest.config.ts thresholds to be 5% below actual numbers (prevents regression while allowing minor fluctuation):
     - If actual statements = 85%, set threshold to 80%
     - If actual branches = 72%, set threshold to 67%
   - Minimum thresholds (floor): statements 70%, branches 55%, functions 65%, lines 70%

5. Apply any quick performance wins found (lazy imports, React.memo). Skip complex refactors — document those as future improvements.
  </action>

    - `npx vitest run --coverage` passes threshold check
    - Heavy dependencies (@react-pdf/renderer, recharts) use dynamic imports or are documented as already lazy
  </acceptance_criteria>
