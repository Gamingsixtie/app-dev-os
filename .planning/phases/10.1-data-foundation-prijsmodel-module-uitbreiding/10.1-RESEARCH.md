# Phase 10.1: Data Foundation -- Prijsmodel & Module-uitbreiding - Research

**Researched:** 2026-03-24
**Domain:** TypeScript data architecture, discriminated unions, pricing model restructuring
**Confidence:** HIGH

## Summary

Phase 10.1 is a pure data-layer restructuring: the current pricing data is scattered across 4 separate files (`default-prices.ts`, `dia-packages.ts`, `jij-license-tiers.ts`, `cito-bundles.ts`) with the MODULE_CATALOG limited to 6 modules. The goal is to (1) expand the module catalog to the full VO product range, (2) define a `PricingStrategy` discriminated union that captures the 4 distinct pricing models (flat, tiered-license, package-bundle, platform+module), and (3) consolidate provider-specific data into `src/data/providers/{cito,dia,jij}.ts` files while keeping backward compatibility with all 18+ consumers.

The codebase already handles provider-specific pricing logic in the store (`applyDynamicJijPrices`, `applyDiaVolumeDiscount`, `applyCitoBundlePrices`) -- this phase does NOT refactor the engine (that is Phase 10.2). This phase creates the correct data structures and migrates existing data into them. The key challenge is maintaining all existing imports and test compatibility during migration.

**Primary recommendation:** Use re-exports from original file locations to maintain backward compatibility. Create provider config files as the single source of truth, then re-export `DEFAULT_PRICES`, `DIA_PACKAGES`, `JIJ_LICENSE_TIERS` etc. from their original paths. This allows incremental migration without breaking 18+ consumers.

### Pattern 1: Discriminated Union for PricingStrategy
**What:** A TypeScript discriminated union that models the 4 distinct pricing approaches
**When to use:** Whenever code needs to understand HOW a provider prices a module
**Example:**
```typescript
// src/models/pricing.ts (additions)

interface FlatPricing {
  type: 'flat';
  pricePerStudent: number;
}

interface TieredLicensePricing {
  type: 'tiered-license';
  tiers: JijLicenseTier[];
  pricePerTest: number; // varies by tier
  magisterFee: number;
}

interface PackageBundlePricing {
  type: 'package-bundle';
  packages: DiaPackage[];
  individualPrices: Map<string, number>;
  volumeDiscountTiers: Array<{ minStudents: number; discountPercent: number }>;
}

interface PlatformModulePricing {
  type: 'platform+module';
  bundles: CitoBundle[];
  contractPeriods: ContractPeriodConfig[];
  individualPrices: Map<string, number>;
}

export type PricingStrategy =
  | FlatPricing
  | TieredLicensePricing
  | PackageBundlePricing
  | PlatformModulePricing;
```

### Pattern 2: Provider Configuration Object
**What:** A single typed configuration per provider containing all pricing data
**When to use:** Provider-specific config files in `src/data/providers/`
**Example:**
```typescript
// src/data/providers/dia.ts
import type { PricingStrategy } from '@/models/pricing';
import type { DiaPackage } from '@/models/dia-packages';

export interface DiaProviderConfig {
  key: 'dia';
  label: 'DIA';
  pricingStrategy: PricingStrategy; // type: 'package-bundle'
  modules: DiaModuleConfig[];       // which modules DIA offers + prices
  packages: DiaPackage[];           // existing DIA_PACKAGES data
  volumeDiscountTiers: typeof DIA_VOLUME_TIERS;
}

// Contains ALL existing data from dia-packages.ts + dia entries from default-prices.ts
export const DIA_CONFIG: DiaProviderConfig = { ... };

// Re-exports for backward compatibility
export const DIA_PACKAGES = DIA_CONFIG.packages;
```

### Pattern 3: Expanded MODULE_CATALOG with Provider Availability
**What:** Module definitions include which providers offer them, with aliases for fuzzy matching
**When to use:** MODULE_CATALOG in `src/models/modules.ts`
**Example:**
```typescript
export interface ModuleDefinition {
  id: string;
  name: string;
  description: string;
  category: ModuleCategory;
  separateLicense: boolean;
  differentiator?: string;
  // NEW fields:
  aliases: string[];                    // For AI intake fuzzy matching
  availableFrom: ProviderKey[];         // Which providers offer this module
}
```

### Pattern 4: Re-Export for Backward Compatibility
**What:** Original file paths become thin re-export wrappers
**When to use:** After migrating data to provider configs
**Example:**
```typescript
// src/data/default-prices.ts (after migration)
// Backward-compatible re-export from provider configurations
import { CITO_CONFIG } from './providers/cito';
import { DIA_CONFIG } from './providers/dia';
import { JIJ_CONFIG } from './providers/jij';
import { SAQI_CONFIG } from './providers/saqi';

export const DEFAULT_PRICES: PriceRecord[] = [
  ...CITO_CONFIG.defaultPrices,
  ...DIA_CONFIG.defaultPrices,
  ...JIJ_CONFIG.defaultPrices,
  ...SAQI_CONFIG.defaultPrices,
];
```

### Anti-Patterns to Avoid
- **Breaking existing imports:** Do NOT rename or move files that have 18+ consumers without re-exports
- **Mixing data and logic:** Provider config files contain DATA only. Calculation logic stays in `src/engine/`
- **Duplicating prices:** Each price value should exist in exactly one place (the provider config). Everything else re-exports
- **Modifying price values:** This phase restructures, it does NOT change any actual euro amounts

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Discriminated union exhaustiveness | Manual if/else chains | TypeScript `switch` with `never` default | Compiler catches missing cases |
| Module availability checking | Custom per-provider logic | `availableFrom` array on ModuleDefinition | Single source of truth |
| Backward compatibility | Updating 18+ import sites | Re-exports from original paths | Zero-risk migration |

## Common Pitfalls

### Pitfall 1: Breaking the Store Recalculation Chain
**What goes wrong:** The price-comparison store has a complex recalculate() chain: DEFAULT_PRICES -> applyDiaVolumeDiscount -> applyDynamicJijPrices -> applyCitoBundlePrices -> calculateComparison. Any change to data shapes breaks this.
**Why it happens:** The store imports directly from 5+ data files. Changing exports breaks the chain.
**How to avoid:** Use re-exports so original import paths continue to work. Test by running `npm run build` after each file change.
**Warning signs:** TypeScript errors in `store.ts`, failing `npx vitest run`.

### Pitfall 2: Circular Imports Between Provider Configs and Models
**What goes wrong:** Provider config files import from models, models import types from provider configs.
**Why it happens:** PricingStrategy type needs DiaPackage/CitoBundle types, but those are currently in separate model files.
**How to avoid:** Keep type definitions in `src/models/`. Provider configs import types from models, never the reverse.
**Warning signs:** "Cannot access before initialization" errors.

### Pitfall 3: Accidentally Changing Price Values During Migration
**What goes wrong:** Copy-paste errors or rounding differences change actual euro amounts.
**Why it happens:** Manual transcription of numeric values between files.
**How to avoid:** Write a migration test that asserts `DEFAULT_PRICES` output is identical before and after restructuring. Use snapshot testing.
**Warning signs:** Failing price comparison tests, different totals in UI.

### Pitfall 4: MODULE_CATALOG ID Changes Breaking References
**What goes wrong:** New module IDs don't match existing references in DEFAULT_PRICES, differentiators, provider-module-content.
**Why it happens:** New modules added with inconsistent ID conventions.
**How to avoid:** Define module IDs in MODULE_CATALOG first, then reference those IDs everywhere. All IDs use kebab-case (existing convention: `rekenwiskunde`, `sociaal-emotioneel`, etc.).
**Warning signs:** Undefined prices for new modules.

### Pitfall 5: Test Snapshot Drift
**What goes wrong:** Existing engine tests hardcode module IDs and price values. New modules or restructured data causes test failures.
**Why it happens:** Tests use `makePrice('rekenwiskunde', 'cito', 4.5)` -- these test helpers create synthetic data, so they won't break. But tests that import `DEFAULT_PRICES` or `DIA_PACKAGES` directly WILL see changes.
**How to avoid:** Identify all tests importing from data files (found: `schijnvoordeel.test.ts`, `cito-bundles.test.ts`, `dia-packages.test.ts`). Ensure re-exports preserve exact values.
**Warning signs:** Test failures in tests you didn't modify.

function getPricingLabel(strategy: PricingStrategy): string {
  switch (strategy.type) {
    case 'flat':
      return `Vaste prijs: ${strategy.pricePerStudent}/lln`;
    case 'tiered-license':
      return 'Licentie + toetsprijs model';

  }

## Module Catalog Analysis

### Current Catalog (6 modules)
1. `rekenwiskunde` -- Reken-Wiskunde (LVS)
2. `nederlands` -- Nederlands (LVS)
3. `engels` -- Engels (LVS)
4. `taalverzorging` -- Taalverzorging Nederlands (overig)
5. `sociaal-emotioneel` -- Sociaal-emotioneel functioneren (overig)
6. `cognitieve-capaciteiten` -- Cognitieve capaciteitentoets (overig)

### What Schools Actually Select
Schools choose which "toetsdomeinen" (testing domains) they want. The 6 current modules represent the core VO toetsdomeinen that Cito offers. DIA and JIJ offer products that map to these same domains (plus some extras like MVT for JIJ).

### Potential Catalog Expansion
Based on the provider-module-content.ts data and DIA webshop:
- **Leer-werkhouding (LWH)**: Cito offers this alongside SEF. Currently bundled in `sociaal-emotioneel` but could be a separate selectable module (Cito charges separately for it).
- **MVT (Frans/Duits/Spaans)**: JIJ offers these, Cito and DIA do not. Could be added as selectable modules for schools using JIJ.
- **DIA oefenmateriaal (Tekstenlab, Spellab, Burgerschap)**: These are add-on products, not core testing modules. Better as optional add-ons in the DIA provider config.
- **Schoolexamens**: JIJ offers kijk-/luistertoetsen as schoolexamen-alternatief. Pricing is separate (schoolExamPrice in JIJ tiers).

### Recommendation for MODULE_CATALOG Expansion
Keep the catalog school-centric (what domains does a school want to test?), NOT provider-centric. Add modules where at least one provider offers a distinct product with its own pricing:

| Potential New Module | Category | Available From | Rationale |
|---------------------|----------|----------------|-----------|
| `leer-werkhouding` | overige-instrumenten | cito | Cito prices SEF and LWH separately (both EUR 3.00). Currently hidden inside sociaal-emotioneel |
| `frans` | overige-instrumenten | jij | JIJ offers distinct ERK-tested French. No Cito/DIA equivalent |
| `duits` | overige-instrumenten | jij | JIJ offers distinct ERK-tested German. No Cito/DIA equivalent |
| `spaans` | overige-instrumenten | jij | JIJ offers distinct ERK-tested Spanish. No Cito/DIA equivalent |

**Note:** The success criteria says "niet slechts 6 modules" and "volledige VO-productcatalogus per aanbieder". The exact expansion should be confirmed during planning based on what the user considers the "complete" catalog. The above 4 additions bring it to 10, which covers all distinct testable domains across providers.

## Open Questions

1. **Exact module expansion scope**
   - What we know: Current catalog has 6 modules. DIA/JIJ offer products mapping to MVT (Frans/Duits/Spaans) and Cito has LWH as separate pricing.
   - What's unclear: Does the user want all possible modules or just the ones with direct price competition?
   - Recommendation: Start with 10 modules (add leer-werkhouding, frans, duits, spaans) -- these represent all distinct testable domains. The planner should note this needs user confirmation.

2. **SAQI provider config**
   - What we know: SAQI has only one module (sociaal-emotioneel) with flat EUR 3.50/student pricing.
   - What's unclear: Whether SAQI warrants its own full provider config file or should remain inline.
   - Recommendation: Create `providers/saqi.ts` for consistency, even if simple. It maintains the 1-provider-1-file pattern.

3. **Leer-werkhouding as separate module vs. bundled**
   - What we know: Cito prices SEF (EUR 3.00) and LWH (EUR 3.00) separately. Current catalog bundles both under `sociaal-emotioneel`.
   - What's unclear: Whether schools select LWH independently or always with SEF.
   - Recommendation: Model as separate module -- schools do choose LWH independently. Plus-bundel includes both.
