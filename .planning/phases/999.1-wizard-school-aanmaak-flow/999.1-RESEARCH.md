# Phase 999.1: Wizard als onderdeel van school-aanmaak flow - Research

**Researched:** 2026-03-25
**Domain:** UX flow refactoring — wizard integration into school creation
**Confidence:** HIGH

## Summary

This phase restructures the user journey so that creating a new school immediately launches the comparison wizard. Currently, `AddSchoolButton.handleManual()` calls `createSchool('Nieuw schoolprofiel')` and navigates to `/scholen/$slug/wizard/1`. The school is created with a generic name and zero profile data — the wizard is technically already the first thing a user sees, but the school name defaults to "Nieuw schoolprofiel" and the wizard feels detached from the creation act.

The goal is to make the wizard the **entry point** of every school conversation — not a separate tab you discover later. This means: (1) the school creation flow should feel like starting the wizard, not "creating an empty record then filling it", and (2) for returning users opening an incomplete school, the wizard should be prominently surfaced rather than hidden behind the tab system.

The existing codebase already has all building blocks: wizard steps with Zod schemas, `createSchool()` in operations.ts, `isComplete`/`completedSteps` tracking in SchoolRecord, and the IncompleteIndicator badge on SchoolCard. The work is primarily UX flow changes and minor routing/component adjustments — no new libraries or architectural changes needed.

**Primary recommendation:** Refactor the school-creation entry point so the user names their school (or picks from list) in a lightweight dialog, then immediately enters the wizard. For incomplete schools, auto-redirect to wizard from dashboard or show a prominent "Profiel voltooien" CTA.

## Architecture Patterns

### Current Flow (AS-IS)
```
SchoolOverviewPage
  -> AddSchoolButton (click "School toevoegen")
    -> createSchool('Nieuw schoolprofiel')
    -> navigate to /scholen/$slug/wizard/1
      -> SchoolLayout (hydrates stores)
        -> WizardPage -> WizardShell (5 steps)
          -> On last step: navigate to /vergelijking or /huidig-vs-cito or /migratie
```

### Proposed Flow (TO-BE)
```
SchoolOverviewPage
  -> AddSchoolButton (click "School toevoegen")
    -> SchoolNameDialog (user enters school name)
      -> createSchool(name)
      -> navigate to /scholen/$slug/wizard/1
        -> WizardShell (5 steps, same as now)

For existing incomplete schools:
  SchoolLayout
    -> if !isComplete: redirect to /wizard/1 OR show "Profiel voltooien" banner
```

### Pattern 1: School Name Collection Before Creation
**What:** Collect school name before calling `createSchool()` instead of using generic default.
**When to use:** Every school creation path (manual + from-list).
**Approach:** The `SchoolPickerDialog` already does this for the "from list" path (calls `handlePickerCreate(name)`). Extend similar UX to the manual path — either inline in AddSchoolButton or via a small dialog.

### Pattern 2: Incomplete School Redirect
**What:** When a user navigates to an incomplete school's dashboard, guide them to finish the wizard.
**When to use:** SchoolLayout or DashboardTab detects `school.isComplete === false`.
**Approach:** Either (a) auto-redirect to wizard, or (b) show a prominent banner/CTA. Option (b) is safer — auto-redirect may surprise users who want to edit other tabs (contacts, prices) before finishing the wizard.

### Pattern 3: Wizard as Onboarding Context
**What:** Update WizardPage header text to feel like "school setup" rather than a standalone tool.
**When to use:** When school has `completedSteps.length === 0` (fresh creation).
**Approach:** Conditional header: "Schoolprofiel instellen" for new schools vs. current "Prijsvergelijking voor uw school" for returning edits.

### Key Files to Modify
```
src/features/school-overview/AddSchoolButton.tsx     # School name dialog
src/components/routing/SchoolLayout.tsx               # Incomplete school handling
src/components/routing/WizardPage.tsx                 # Context-aware header
src/features/school-profile/tabs/DashboardTab.tsx     # "Profiel voltooien" CTA
src/features/school-overview/SchoolCard.tsx            # Click behavior for incomplete schools
```

### Anti-Patterns to Avoid
- **Auto-redirect without escape:** Never force-redirect to wizard with no way to access other tabs. Users may want to add contacts or upload a schoolplan before completing the wizard.
- **Creating school records without user input:** The current `createSchool('Nieuw schoolprofiel')` pattern leaves orphan records if user abandons. Collect name first, then create.
- **Duplicating wizard logic:** Do NOT create a separate "creation wizard" — reuse the existing 5-step wizard. The change is about the entry flow, not the wizard itself.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| School name input dialog | Custom modal from scratch | Extend existing SchoolPickerDialog pattern | Consistent UX, already handles name collection |
| URL-based step tracking | Custom step state manager | Existing `completedSteps` + `currentStep` in store | Already working, persisted to Supabase |
| Incomplete school detection | Custom query | `school.isComplete` field from SchoolRecord | Already tracked, already shown via IncompleteIndicator |

## Common Pitfalls

### Pitfall 1: Orphaned Empty School Records
**What goes wrong:** User clicks "School toevoegen", school is created in Supabase, but user navigates away before entering any wizard data.
**Why it happens:** `createSchool()` is called before any user commitment (name entry).
**How to avoid:** Collect school name in a dialog BEFORE calling `createSchool()`. Only create the DB record after user confirms name. This is already how `SchoolPickerDialog` works for the "from list" path.
**Warning signs:** Many schools in the overview with name "Nieuw schoolprofiel" and `isComplete: false`.

### Pitfall 2: SchoolLayout Hydration Race
**What goes wrong:** If you redirect from SchoolLayout to wizard before `useSchool(slug)` resolves, the store may not be hydrated.
**Why it happens:** SchoolLayout hydrates stores in a `useEffect` that depends on `school` data. Redirect must happen AFTER hydration.
**How to avoid:** Put redirect/CTA logic in DashboardTab (which renders after layout hydration), not in SchoolLayout itself.
**Warning signs:** Wizard shows empty fields for a school that has data.

### Pitfall 3: Breaking the "From List" Import Path
**What goes wrong:** `SchoolListImportDialog` -> `SchoolPickerDialog` -> `handlePickerCreate(name)` already creates schools with a real name. Changes to AddSchoolButton must not break this path.
**Why it happens:** AddSchoolButton has two paths: manual and from-list. They share some logic but diverge in name collection.
**How to avoid:** Test both paths after changes. The from-list path already collects name — only the manual path needs modification.
**Warning signs:** "Kies uit schoolenlijst" flow stops working.

### Pitfall 4: Slug Change on Step 1
**What goes wrong:** WizardStep1 already allows renaming the school AND updates the slug via `uniqueSlug()`. If you add a name dialog before creation, the slug may change in step 1, breaking the URL.
**Why it happens:** Step 1 calls `updateSchoolData(activeSchoolId, { slug: newSlug, name: data.schoolName })` when name changes.
**How to avoid:** After Step 1 changes the slug, WizardShell must navigate to the new slug URL. This is already handled — verify it still works when name is pre-set from the creation dialog.
**Warning signs:** 404 after renaming school in step 1.

## Open Questions

1. **Should incomplete schools auto-redirect to wizard or show a CTA banner?**
   - What we know: Auto-redirect is simpler but may block access to other tabs. CTA banner is less disruptive.
   - Recommendation: CTA banner on DashboardTab with prominent styling. Users can still access other tabs.

2. **Should the school name dialog be a full modal or inline?**
   - What we know: SchoolPickerDialog is already a modal pattern. Could also do inline expansion in AddSchoolButton.
   - Recommendation: Small modal dialog (consistent with existing SchoolPickerDialog pattern). Quick input field + "Start" button.

3. **What about existing schools with name "Nieuw schoolprofiel"?**
   - What we know: There may be existing schools in production with this default name.
   - Recommendation: No migration needed. These schools will naturally be renamed when users go through Step 1 (which already has the school name field).
